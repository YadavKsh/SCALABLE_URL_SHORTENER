## 🧱 Production Architecture

Locally, everything ran on your laptop. In production, each component runs as a separate managed service:

```
User's browser
      │
      │  HTTPS request
      ▼
┌─────────────────────────┐
│   Netlify               │  ← serves the frontend (HTML, CSS, JS)
│   Frontend              │    static files, global CDN, free tier
└────────────┬────────────┘
             │
             │  POST /shorten  or  GET /{shortCode}
             │  (HTTPS to Render)
             ▼
┌─────────────────────────┐
│   Render                │  ← runs the Spring Boot backend
│   Backend (Docker)      │    inside a Docker container
└──────┬──────────────────┘
       │                  │
       │                  │
       ▼                  ▼
┌──────────────┐   ┌──────────────────────┐
│   Render     │   │   Upstash            │
│   PostgreSQL │   │   Redis              │
│              │   │                      │
│  Permanent   │   │  Cache layer         │
│  storage     │   │  (in-memory, fast)   │
└──────────────┘   └──────────────────────┘
```

**The request flow in production is identical to local development** — the only difference is the addresses. Instead of `localhost:8081`, the backend lives at `https://your-app.onrender.com`. The architecture, caching strategy, and database logic are unchanged.

| Service | What it hosts | Why this choice |
|---|---|---|
| **Netlify** | Frontend (HTML/CSS/JS) | Free tier, drag-and-drop deploy, global CDN, HTTPS automatic |
| **Render** | Spring Boot backend | Free tier supports Docker, auto-deploys from GitHub, managed TLS |
| **Render PostgreSQL** | Database | Managed, automatic backups, same platform as backend (low latency) |
| **Upstash Redis** | Cache | Only Redis provider with a generous free tier; serverless, pay-per-request |

All four services have free tiers sufficient for a portfolio project. No credit card required for the basic plans.

## 🗄️ Step 1 — PostgreSQL on Render

**Do this first** — you need the database credentials before configuring the backend.

### Creating the Database

1. Go to [render.com](https://render.com) and sign in
2. Click **New +** → **PostgreSQL**
3. Fill in the form:
    - **Name:** `url-shortener-db` (any name, just for your reference)
    - **Database:** `url_shortener`
    - **User:** leave as auto-generated or set your own
    - **Region:** choose the one closest to your users
    - **Plan:** Free
4. Click **Create Database**

Render will take about 60 seconds to provision the database.

### Saving Your Credentials

Once created, scroll down to the **Connections** section. You'll see:

```
Hostname:          oregon-postgres.render.com
Port:              5432
Database:          url_shortener
Username:          url_shortener_user
Password:          <auto-generated>
Internal Database URL: postgres://...   ← use this for the backend on Render
External Database URL: postgres://...   ← use this to connect from your laptop
```

**Save all of these** — you'll need them in Step 5 when setting environment variables on Render.

The **Internal Database URL** is what you'll use for the backend environment variable. Since both the backend and database run on Render, using the internal URL keeps traffic within Render's private network — faster and more secure than the external URL.

> **📌 Important:** The free tier PostgreSQL on Render **expires after 90 days**. For a long-lived portfolio project, consider Supabase or Neon as free alternatives with no expiry. The connection string format is identical — just swap the URL.

## ⚡ Step 2 — Redis on Upstash

Upstash is the only cloud Redis provider with a genuinely free tier (10,000 commands/day). It also requires SSL, which is handled by a single configuration line.

### Creating the Redis Instance

1. Go to [console.upstash.com](https://console.upstash.com) and sign in
2. Click **Create Database**
3. Fill in the form:
    - **Name:** `url-shortener-cache`
    - **Type:** Regional
    - **Region:** same as your Render backend
    - **Enable Eviction:** on (so Redis automatically removes old entries when memory fills up)
4. Click **Create**

### Saving Your Credentials

Once created, click on your database and go to the **Details** tab:

```
Endpoint:   your-instance.upstash.io
Port:       6379
Password:   <auto-generated token>
```

You also need to enable SSL — Upstash **requires** encrypted connections. Add this to `application.properties`:

```properties
spring.data.redis.ssl.enabled=true
```

Without this line, your backend will fail to connect to Upstash with a `connection refused` or `SSL handshake failed` error.

## ⚙️ Step 3 — Preparing the Backend for Production

The backend needs three changes before it's production-ready. These changes make the app read configuration from environment variables instead of hardcoded values, and generate URLs dynamically instead of assuming `localhost`.

### Change 1 — Externalise Configuration

Locally, `application.properties` had hardcoded values:

```properties
# LOCAL (development only — never commit passwords to GitHub)
spring.datasource.url=jdbc:postgresql://localhost:5432/url_shortener
spring.datasource.username=postgres
spring.datasource.password=your_password
spring.data.redis.host=localhost
spring.data.redis.port=6379
```

In production, credentials live in **environment variables** — not in code. Update `application.properties` to read from environment variables using `${}` syntax:

```properties
# ── Server ───────────────────────────────────────────────────
server.port=${PORT:8081}
# Reads the PORT environment variable. Falls back to 8081 if not set.
# Render sets PORT automatically (usually 10000).

# ── PostgreSQL ────────────────────────────────────────────────
spring.datasource.url=${DB_URL}
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=false

# ── Redis (Upstash) ───────────────────────────────────────────
spring.data.redis.host=${REDIS_HOST}
spring.data.redis.port=${REDIS_PORT}
spring.data.redis.password=${REDIS_PASSWORD}
spring.data.redis.ssl.enabled=true
```

The `${VARIABLE_NAME}` syntax tells Spring: *"look for an environment variable with this name at startup."* If the variable isn't set, Spring throws an error immediately — which is better than silently connecting to the wrong database.

`${PORT:8081}` is slightly different — the `:8081` after the colon is the **default value**. This means: use `PORT` if it exists, otherwise fall back to `8081`. Useful because Render sets `PORT` automatically, but your local machine doesn't.

> **📌 Never commit real credentials to GitHub.** Once a password is in a public repository's git history, it should be considered compromised — even if you delete the file later. The `${ENV_VAR}` pattern ensures credentials live only in the deployment platform's environment settings, never in your code.

---

### Change 2 — Dynamic URL Generation

Locally, the service hardcoded the base URL:

```java
// ❌ Hardcoded — breaks in production
String shortUrl = "http://localhost:8081/" + shortCode;
```

In production, the domain is `https://your-app.onrender.com` — not `localhost:8081`. Hardcoding the localhost URL means every short link the production system generates will point to a non-existent address.

The fix uses Spring's `ServletUriComponentsBuilder` to build the base URL dynamically from the current request:

```java
import org.springframework.web.servlet.support.ServletUriComponentsBuilder;

// In the controller's shortenUrl() method:
String baseUrl = ServletUriComponentsBuilder
        .fromCurrentContextPath()   // reads the current request's scheme + host + port
        .build()
        .toUriString();

String shortUrl = baseUrl + "/" + shortCode;
```

`fromCurrentContextPath()` inspects the incoming HTTP request and extracts:
- The **scheme** (`http` or `https`)
- The **host** (`localhost` or `your-app.onrender.com`)
- The **port** (`:8081` locally, nothing in production since HTTPS uses 443)

When a request arrives at `https://your-app.onrender.com/shorten`, `fromCurrentContextPath()` returns `https://your-app.onrender.com`. The generated short URL becomes `https://your-app.onrender.com/abc123` — a real, clickable link.

Locally the same code returns `http://localhost:8081` — nothing breaks in development either.

---

### Change 3 — Update CORS for the Production Frontend

The controller currently allows all origins:

```java
@CrossOrigin(origins = "*")
```

This is fine for development. For production, restrict it to your actual Netlify URL:

```java
@CrossOrigin(origins = {
    "http://localhost:3000",                        // local development
    "https://your-app-name.netlify.app"             // production frontend
})
```

Replace `your-app-name` with your actual Netlify subdomain. You can update this after deploying to Netlify once you know the URL.

## 🐳 Step 4 — Dockerising the Backend

Docker packages your entire application — the JDK, Maven, your code, and all dependencies — into a single self-contained image. Render runs this image exactly as-is, with no setup required on their servers.

### The Dockerfile

Create a file named `Dockerfile` inside the `backend/` folder:

```dockerfile
# ── Stage: Base image ─────────────────────────────────────────
FROM eclipse-temurin:21-jdk-jammy
# eclipse-temurin is the official OpenJDK distribution
# 21 = Java 21 (LTS)
# jammy = Ubuntu 22.04 base OS (stable, widely supported)

# ── Set working directory ─────────────────────────────────────
WORKDIR /app
# Creates the /app directory inside the container and sets it
# as the working directory for all subsequent commands.
# Think of it as: cd /app inside the container.

# ── Copy the entire project into the container ─────────────────
COPY . .
# Copies everything from your project folder (on your machine)
# into /app (inside the container).

# ── Move into the backend subfolder ───────────────────────────
WORKDIR /app/backend
# If your repo structure is:
#   /app/
#     frontend/
#     backend/   ← Spring Boot lives here
# This changes the working directory to /app/backend.

# ── Make the Maven Wrapper executable ─────────────────────────
RUN chmod +x mvnw
# On Linux (where Docker containers run), shell scripts must be
# explicitly marked as executable. Without this, the next command
# fails with "permission denied".

# ── Build the application ─────────────────────────────────────
RUN ./mvnw clean package -DskipTests
# clean    → delete any previous build output
# package  → compile the code and package it into a JAR file
# -DskipTests → skip running tests during the build
#               (tests require a DB connection, which isn't available
#               at Docker build time)
# Output: target/backend-0.0.1-SNAPSHOT.jar

# ── Tell Docker which port the app listens on ─────────────────
EXPOSE 8081
# Documents that the application listens on port 8081.
# This is documentation only — Render maps its own PORT to this.

# ── The command that runs when the container starts ───────────
CMD ["java", "-jar", "target/backend-0.0.1-SNAPSHOT.jar"]
# Starts the Spring Boot application by executing the JAR file.
# This is the equivalent of running: java -jar target/backend-0.0.1-SNAPSHOT.jar
# in your terminal.
```

### Testing the Docker Build Locally

Before pushing to Render, verify the image builds on your machine:

```bash
# Navigate to the root of your project (not inside backend/)
cd scalable-url-shortener

# Build the Docker image and tag it with a name
docker build -f backend/Dockerfile -t url-shortener-backend .

# Run the container locally to test it
docker run -p 8081:8081 \
  -e DB_URL=jdbc:postgresql://localhost:5432/url_shortener \
  -e DB_USERNAME=postgres \
  -e DB_PASSWORD=your_password \
  -e REDIS_HOST=localhost \
  -e REDIS_PORT=6379 \
  -e REDIS_PASSWORD="" \
  url-shortener-backend
```

If the Spring Boot startup logs appear and the app runs — the Docker build is correct.

## 🌐 Step 5 — Deploying the Backend to Render

### Connect Your GitHub Repository

1. Push your entire project to GitHub (including the `Dockerfile`)
2. Go to [render.com](https://render.com) → **New +** → **Web Service**
3. Connect your GitHub account and select your repository
4. Configure the service:

| Setting | Value |
|---|---|
| **Name** | `url-shortener-backend` |
| **Environment** | `Docker` |
| **Dockerfile Path** | `backend/Dockerfile` |
| **Region** | Same as your PostgreSQL instance |
| **Plan** | Free |

### Set Environment Variables

In the **Environment** section of the Render service, add each variable:

```
PORT              = 10000
DB_URL            = jdbc:postgresql://<internal-host>:5432/url_shortener
DB_USERNAME       = <from Render PostgreSQL credentials>
DB_PASSWORD       = <from Render PostgreSQL credentials>
REDIS_HOST        = <your-instance>.upstash.io
REDIS_PORT        = 6379
REDIS_PASSWORD    = <your Upstash token>
```

Use the **Internal Database URL** from Render PostgreSQL for `DB_URL`. Copy the hostname from that URL and construct it as shown above.

> **📌 Why `PORT=10000`?** Render assigns port `10000` to free-tier web services internally and routes external HTTPS traffic to it. Your `application.properties` reads `${PORT:8081}` — so in production it uses `10000`, locally it defaults to `8081`.

### Deploy

Click **Create Web Service**. Render will:
1. Clone your repository
2. Build the Docker image (runs `mvnw clean package`)
3. Start the container
4. Assign a public URL: `https://url-shortener-backend-xxxx.onrender.com`

The first build takes 3–5 minutes. Subsequent deployments (triggered by pushing to GitHub) are faster.

**Confirming successful startup** — in the Render logs, look for:

```
Started BackendApplication in 8.234 seconds
Tomcat started on port(s): 10000 (http)
```

If you see this, the backend is live. Test it:

```bash
curl -X POST https://your-app.onrender.com/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://google.com"}'
```

Expected response:
```json
{
  "shortUrl": "https://your-app.onrender.com/abc123",
  "shortCode": "abc123",
  "originalUrl": "https://google.com"
}
```

> **📌 Free tier cold starts:** Render's free tier spins down containers after 15 minutes of inactivity. The first request after a period of inactivity can take 30–60 seconds while the container starts back up. Subsequent requests are fast. This is a free-tier limitation — paid tiers keep the service always running.

## 🌍 Step 6 — Deploying the Frontend to Netlify

### Update the API URL

Before deploying, update `js/app.js` to point to your production backend:

```js
// Before (local development):
const API_URL = "http://localhost:8081/shorten";

// After (production):
const API_URL = "https://your-app.onrender.com/shorten";
```

Replace `your-app.onrender.com` with the actual URL Render assigned to your backend.

### Deploy via Drag and Drop

1. Go to [netlify.com](https://netlify.com) and sign in
2. On the dashboard, find the **drag and drop** zone at the bottom: *"Want to deploy a new site without connecting to Git? Drag and drop your site output folder here"*
3. Drag your entire `frontend/` folder into that zone
4. Netlify deploys in under 30 seconds and assigns a URL like `https://random-name-123.netlify.app`

### Optional — Custom Domain or Site Name

In Netlify's **Site Settings → General → Site Details**, you can change the random subdomain to something readable: `https://url-shortener-yourname.netlify.app`.

### Confirming the Frontend Works

Open your Netlify URL in a browser. Paste a long URL and click **Shorten**. If you get a short URL back — all four components are connected and working.

## 🧪 Step 7 — Verifying the Full System

Run through this checklist after deployment:

### Database Check

Connect to your Render PostgreSQL from your terminal using the External Database URL:

```bash
psql "postgres://username:password@host:5432/url_shortener"

# Check that the table was created by Hibernate:
\dt

# Expected output:
#          List of relations
#  Schema │    Name      │ Type  │ Owner
# ────────┼──────────────┼───────┼───────
#  public │ url          │ table │ ...

# Check that data is being stored:
SELECT * FROM url;
```

### Redis Cache Check

After making a redirect request, check the Render backend logs:

**First request (cache miss):**
```
💾 Fetched from DB and cached
```

**Second request to the same short URL (cache hit):**
```
⚡ Served from Redis
```

If you only ever see the first message and never the second, Redis isn't connected — check the `REDIS_HOST`, `REDIS_PORT`, and `REDIS_PASSWORD` environment variables on Render, and confirm `spring.data.redis.ssl.enabled=true` is in `application.properties`.

### Persistence Check

1. Create a short URL via the frontend
2. Go to Render → your backend service → **Manual Deploy** → restart the service
3. After restart, open the short URL in a browser

If it still redirects correctly — data is persisted in PostgreSQL, not just in memory. ✅

## 🔄 How the Production System Behaves

### Creating a Short URL (`POST /shorten`)

```
1. User pastes URL into frontend, clicks Shorten
         ↓
2. Frontend sends POST to https://your-app.onrender.com/shorten
         ↓
3. Backend validates the URL
         ↓
4. Service checks PostgreSQL: does this URL already exist?
   └─ Yes → return existing short code (no duplicate stored)
   └─ No  → generate new Base62 code, check for collision
         ↓
5. Save new mapping to Render PostgreSQL
         ↓
6. Immediately cache in Upstash Redis
         ↓
7. Backend builds short URL using ServletUriComponentsBuilder:
   "https://your-app.onrender.com/abc123"
         ↓
8. Response sent to frontend, displayed to user
```

### Redirecting (`GET /{shortCode}`)

```
1. User clicks short link:  https://your-app.onrender.com/abc123
         ↓
2. Backend checks Upstash Redis for "abc123"
   └─ Cache HIT  → return original URL immediately ⚡ (sub-millisecond)
   └─ Cache MISS → query Render PostgreSQL
         ↓ (on miss)
3. PostgreSQL returns original URL
         ↓
4. Store result in Redis for next time
         ↓
5. Backend sends HTTP 302 + Location: https://original-url.com
         ↓
6. Browser follows redirect — user lands on original page
```

## 🔧 Troubleshooting Common Deployment Issues

| Symptom | Likely Cause | Fix |
|---|---|---|
| Backend logs show `DB connection refused` | Wrong `DB_URL` environment variable | Double-check the Internal Database URL from Render PostgreSQL. Make sure you're using the **internal** URL, not the external one |
| `SSL handshake failed` connecting to Redis | Missing SSL config | Add `spring.data.redis.ssl.enabled=true` to `application.properties` |
| Short URLs point to `localhost:8081` in production | `ServletUriComponentsBuilder` not used | Replace the hardcoded base URL string with `ServletUriComponentsBuilder.fromCurrentContextPath()` |
| Frontend shows CORS error in browser console | `@CrossOrigin` doesn't include Netlify URL | Add your Netlify URL to the `origins` list in `@CrossOrigin` on the controller |
| Render build fails at `./mvnw clean package` | `mvnw` not executable | Add `RUN chmod +x mvnw` in the Dockerfile **before** the `RUN ./mvnw` line |
| First request after inactivity takes 60 seconds | Render free tier cold start | Expected behaviour on the free plan. The container was spun down — it restarts on the first incoming request |
| `PORT already in use` on Render | App binds to `8081` but Render sends traffic to `10000` | Make sure `server.port=${PORT:8081}` is in `application.properties` — Render sets `PORT=10000` |
| Redirect works locally but 404 in production | Environment variable `DB_URL` missing or malformed | Check Render's environment variable panel — a missing variable causes the app to start but queries to fail |
